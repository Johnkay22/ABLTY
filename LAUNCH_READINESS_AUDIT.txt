2. CRITICAL ISSUES

--------------------------------------------------------------------------------
C-1 — Stripe checkout URL is the TEST URL (hardcoded)
--------------------------------------------------------------------------------
Priority: Critical
Category: Payment / Subscription / Gating

Issue:
The "Upgrade to Premium" button sends users to a Stripe TEST mode Payment Link.

Why it matters:
Real cards are rejected in test mode, test cards do not produce real revenue,
and webhooks will not be sent from your live Stripe account. Premium will
appear to "work" in QA but no one can actually pay.

Where it appears:
app.html  handleUpgradeCTA()  lines 8104–8123.

Evidence:
  const checkoutUrl = email
    ? ('https://buy.stripe.com/test_5kQfZhe6OfNg5On3p78g000' + '?prefilled_email=' + encodeURIComponent(email))
    : 'https://buy.stripe.com/test_5kQfZhe6OfNg5On3p78g000';

Recommended fix:
Swap to a live-mode Payment Link (or a server-created Checkout Session), and
add a build/env-time guard so test URLs cannot ship to production.

Implementation caution:
The new URL must point at the matching live-mode Product/Price, and the live
webhook signing secret must be configured on the worker. Switching modes
without updating STRIPE_WEBHOOK_SECRET will silently break upgrades.

Confidence: 99%.

C-2 — Stripe checkout does not pass client_reference_id or metadata.supabase_user_id
--------------------------------------------------------------------------------
Priority: Critical
Category: Payment / Account mapping

Issue:
The only signal sent to Stripe is prefilled_email. The worker's
resolveSupabaseUserIdFromStripe() therefore relies on (a) UUID metadata that
never gets set, then (b) stripe_customer: KV mapping that only exists after a
prior event, then (c) email-based admin lookup.

Why it matters:
If the user types a different email at Stripe than their account email (very
common), or signs up later, the webhook silently no-ops (`if (!userId) return;`)
and they never get upgraded. The Payment Link cannot carry client_reference_id
unless you pass ?client_reference_id=... in the URL.

Where it appears:
app.html 8115–8118; ablty-worker.js handleStripeWebhook,
processStripeTierEvent, resolveSupabaseUserIdFromStripe (lines 269–392).

Evidence:
  const targetTier = eventType === 'customer.subscription.deleted' ? 'free' : 'premium';
  const userId = await resolveSupabaseUserIdFromStripe(event, env);
  if (!userId) return;

Recommended fix:
Append client_reference_id=<supabase_user_id> to the Payment Link URL (or move
to a server-created Checkout Session that sets client_reference_id and
metadata.supabase_user_id). Add structured logging when userId cannot be
resolved and queue manual reconciliation.

Implementation caution:
Existing customers paying via email-only flow already exist (test mode) — the
email fallback must remain. Do not log raw access tokens.

Confidence: 95%.

C-3 — findSupabaseUserIdByEmail likely uses the wrong query parameter
--------------------------------------------------------------------------------
Priority: Critical
Category: Payment / Backend

Issue:
The worker calls GET /auth/v1/admin/users?filter=<email>, but the Supabase
Admin API expects ?email=<email> (or paginates and you filter client-side).
?filter is undocumented for that endpoint and may return all users or no users
depending on Supabase version.

Why it matters:
Combined with C-2, the email fallback for upgrades may not match at all in
production, so Stripe webhook events arrive but the user is never flipped to
premium. The webhook still returns 200 ("received: true").

Where it appears:
ablty-worker.js findSupabaseUserIdByEmail (370–392).

Evidence:
  const u = new URL(env.SUPABASE_URL + '/auth/v1/admin/users');
  u.searchParams.set('filter', needle);
  u.searchParams.set('per_page', '10');

Recommended fix:
Verify against current Supabase docs and use the supported parameter
(email=<lowercased> or paginate and match). Add a unit test that hits a real
Supabase project and confirms the lookup works.

Implementation caution:
Admin keys must never reach the browser. Don't expose this lookup to client
routes.

Confidence: 80% (depends on Supabase Auth version).

C-4 — Premium gating is purely client-side for several "premium" features
--------------------------------------------------------------------------------
Priority: Critical
Category: Auth / Gating / Security

Issue:
getCurrentTier() reads localStorage.getItem('ablty_tier'). The user can set
localStorage.ablty_tier = 'premium' in DevTools and unlock:
  - Data export ("Premium" feature), including dream entries, RV sketches, etc.
  - Full analytics history (>7 days)
  - "Community RV", "leaderboards", "ability_profile" UI paths

Why it matters:
Anyone can bypass paywalled UI from a console. The RV daily limit and the
Stripe upgrade are server-checked, but most other premium UX gates are not.
Lost revenue and brand integrity issue if this gets shared on Reddit/Discord.

Where it appears:
app.html lines 7775–7872 (getCurrentTier, canAccess, TIER_ACCESS), 10409
(export), 9421 (Insights cutoff), 11064–11086 (analytics log cutoff).

Evidence:
  function getCurrentTier() {
    if (!isLoggedIn()) return 'guest';
    return localStorage.getItem('ablty_tier') || 'free';
  }

Recommended fix:
For features that touch the cloud (export, full history, dream insights past 7
days), enforce tier server-side (RLS policy or a worker endpoint that re-reads
profiles.tier from Supabase using the service key). Treat the client-side gate
as cosmetic only.

Implementation caution:
Local-only features (RC pager, WBTB timer) are fine; do not over-gate UX.
Server enforcement must still allow Stripe to flip tier in <1 minute (today
the worker caches tier 5 minutes).

Confidence: 90%.

C-5 — handleGrade retry path (target_id) bypasses the RV daily limit
--------------------------------------------------------------------------------
Priority: Critical
Category: API / Gating

Issue:
/rv-assign enforces the free-tier 2/day limit. But /grade accepts either
assignment_id OR target_id. The target_id retry branch does not consume an
assignment and /grade itself does not increment any daily counter — so a user
who knows any valid target_id (T001..T036) can call /grade repeatedly without
ever requesting a new assignment.

Why it matters:
Premium pressure for unlimited RV is undermined; LLM costs spike. Cost-of-goods
on Gemini grading is real money.

Where it appears:
ablty-worker.js lines 182–227 (rate limiting), 850–961 (handleGrade, retry
branch 890–898).

Evidence:
  } else if (retryTargetId) {
    // Retry flow: look up target directly by ID (assignment expired or consumed)
    target = RV_TARGET_POOL.find((t) => t.id === retryTargetId);
    if (!target) {
      return json({ error: 'Target not found' }, 400, origin);
    }
  } else {
    return json({ error: 'Missing assignment_id or target_id' }, 400, origin);
  }

Recommended fix:
Increment the rvdaily: counter inside /grade when the retry branch is used by
a non-premium user, OR require the retry path to also include a valid
(consumed) assignment id, OR require auth and rate-limit per user.

Implementation caution:
Don't double-count legitimate retries after a transient Gemini failure (where
the user did consume an assignment but never got a result back). Consider
tracking "billable grades per day" rather than per-assignment.

Confidence: 90%.

C-6 — Server-side authoritative gate on RV is per-IP, not per-user
--------------------------------------------------------------------------------
Priority: Critical
Category: Gating / Multi-tenancy

Issue:
Both the per-day rate limit (ratelimit:rvassign:) and the free-tier 2/day cap
(rvdaily:) key on CF-Connecting-IP. Premium check happens via bearer token,
but unsigned-in users on shared NAT/IPv4 (universities, schools, family Wi-Fi,
mobile carriers, corporate proxies) get blocked because somebody else used
both slots.

Why it matters:
Real users will be blocked from any RV at all on day 1 from popular networks.
Conversely, a single user moving between Wi-Fi and cellular doubles their
quota.

Where it appears:
ablty-worker.js lines 204–224.

Evidence:
  if (!isPremium) {
    const dailyKey = 'rvdaily:' + ip + ':' + new Date().toISOString().slice(0, 10);
    const dailyCount = parseInt(await env.ABLTY_KV.get(dailyKey) || '0', 10);
    if (dailyCount >= 2) {
      return json({ error: 'daily_limit', message: 'Free tier allows 2 RV sessions per day. Upgrade to Premium for unlimited sessions.' }, 429, origin);
    }
    await env.ABLTY_KV.put(dailyKey, String(dailyCount + 1), { expirationTtl: 86400 });
  }

Recommended fix:
Require authentication on /rv-assign (already partly there) and key the daily
counter on user.id for free users; only fall back to IP-based for guest users,
with a higher cap.

Implementation caution:
Guests have no user id; keep an IP fallback but warn the user clearly when the
cap is the network's not theirs. Premium check today silently treats
unauthenticated requests as "not premium" — keep that behavior.

Confidence: 95%.

C-7 — wbtb-alarms is a single KV object that races and can drop other users' alarms
--------------------------------------------------------------------------------
Priority: Critical
Category: Backend reliability / Sleep alarm

Issue:
handleWBTBSchedule, handleWBTBCancel, and runWBTBCron all do read-modify-write
on the single key wbtb-alarms. KV writes are eventually consistent and there's
no compare-and-swap. Two users scheduling within the same edge cache window
can overwrite each other's alarm.

Why it matters:
WBTB users wake up expecting the alarm. A dropped alarm = no wake-up at the
target time. This is the most user-trust-sensitive feature in the app.

Where it appears:
ablty-worker.js 1086–1196.

Evidence:
  const raw = await env.ABLTY_KV.get('wbtb-alarms');
  const alarms = raw ? JSON.parse(raw) : {};
  alarms[id] = { ... };
  alarms[id + '-return'] = { ... };
  const ttlSeconds = Math.ceil(duration * 3600) + 7200;
  await env.ABLTY_KV.put('wbtb-alarms', JSON.stringify(alarms), { expirationTtl: ttlSeconds });

Recommended fix:
Store each alarm as its own KV key (wbtb:<id> and wbtb:<id>-return) and have
the cron either (a) use list({ prefix: 'wbtb:' }) (with awareness of list-op
quota), or (b) maintain a wbtb-index parallel to sub-index. Avoid concurrent
rewrites of a shared map. Optionally migrate to a Durable Object or D1 if
reliability matters more than KV cost.

Implementation caution:
Migrating storage shape needs a one-time backfill so currently-scheduled
alarms still fire. The TTL also blows away the whole map today — per-key TTL
is per-alarm naturally.

Confidence: 95%.


--------------------------------------------------------------------------------
C-8 — /wbtb-schedule is unauthenticated; trusts client-supplied push endpoint and ID
--------------------------------------------------------------------------------
Priority: Critical
Category: Security / Notifications

Issue:
Anyone who has someone else's push endpoint (a long but not secret-by-design
string) can schedule or cancel WBTB pushes to them. There's no signature or
auth tying the endpoint to the calling user.

Why it matters:
Targeted harassment vector (spammed wake-up alarms in the middle of the
night). Even without targeting, the worker accepts any well-formed payload
and will encrypt+sign it with VAPID. Cancel is also unauthenticated — a
user's alarm can be cancelled by anyone who reproduces their endpoint ID.

Where it appears:
ablty-worker.js handleWBTBSchedule 1086–1125; handleWBTBCancel 1127–1150.

Evidence:
No bearer-token check; only an IP-based rate limit at lines 238–254.

Recommended fix:
Require a Supabase access token and store the user id with the alarm so a
user can only schedule/cancel their own. Cross-reference the endpoint to the
user's stored sub: record.

Implementation caution:
Push endpoints stored in KV sub: need a user_id link before this can be
enforced for legacy subscriptions; backfill on the next sign-in.

Confidence: 90%.

C-9 — User-typed content rendered via innerHTML (XSS-leaning patterns)
--------------------------------------------------------------------------------
Priority: High → Critical for stored XSS
Category: Security / Frontend

Issue:
Many places interpolate user data straight into innerHTML. The risky cases
include:
  - renderResults impressions:
      notesCard.innerHTML = `...<div ...>${notes}</div>`  (line 6886, no escape).
  - buildCombinedLog session card: title, label, thumb URL, ${s.notes} slice
    without escape (11137–11150).

Why it matters:
A user could store HTML/JS in their own impression notes; if those notes ever
appear on a screen another user can see (admin view, exports, future "share
my session"), it becomes a stored-XSS pivot. Even self-only, a paste from the
clipboard with HTML can break layout.

Where it appears:
app.html 6886, 11140–11150 (and several other innerHTML = calls — 86 total
per grep, most are template strings of trusted content but should be audited).

Evidence:
  ${s.type === 'rv' && s.notes && s.notes.trim() ? `<div ... >${s.notes.slice(0,60)}${s.notes.length>60?'..':''}</div>` : ''}

Recommended fix:
Run all user-supplied strings (notes, dream body, dream tags, label, target
labels) through escapeHtml() before interpolation. The helper already exists
(8903–8910); use it consistently.

Implementation caution:
escapeHtml is currently used in dream code but not in RV/Zener/PP/TS render
paths. Don't accidentally escape strings that are intentionally HTML (e.g.,
SVG icons stored as static strings).

Confidence: 95% (vulnerability exists; impact is currently single-user but
compounds the moment any social/admin view is added).

C-10 — Worker has no production observability or alerts beyond console.log
--------------------------------------------------------------------------------
Priority: Critical for launch
Category: Analytics / Admin visibility

Issue:
The only signal on /grade failures, Stripe webhook resolution failures, and
push delivery failures is console.log/console.warn. There is no metric,
dashboard, no error budget. No product analytics on the client (no Plausible,
PostHog, Mixpanel, etc.).

Why it matters:
On launch day you will not be able to answer "how many people tried to
upgrade?", "how many checkout sessions completed?", "how many pushes failed?",
"how many RV sessions saw grading_failed?". Almost no way to triage user
complaints.

Where it appears:
ablty-worker.js throughout; no analytics SDK loaded in app.html (greps for
analytics SDKs return nothing).

Recommended fix:
Add a privacy-respecting product analytics library and instrument: signup,
login, paywall-shown, paywall-CTA-clicked, checkout-redirect,
premium-activated (from webhook), session-completed-per-module,
push-permission-granted/denied, grading-failed. Pipe Cloudflare Worker logs
into a real log sink with alerts on 5xx.

Implementation caution:
Privacy policy says "we do not use tracking cookies, advertising cookies, or
third-party analytics cookies" — pick an analytics tool that fits that
promise (cookieless, EU-hosted), and update the policy if needed.

Confidence: 99%.

H-1 — Stripe webhook missing important events
--------------------------------------------------------------------------------
Priority: High
Category: Payment

Issue:
STRIPE_TIER_EVENTS only includes customer.subscription.created, invoice.paid,
customer.subscription.deleted, checkout.session.completed. It misses
customer.subscription.updated, invoice.payment_failed, and trial-related

Why it matters:

Handle customer.subscription.updated and derive tier from obj.status

Use customer.subscription.deleted as the authoritative downgrade.

H-3 — Email confirmation flow loses legal acceptance metadata across devices
--------------------------------------------------------------------------------
Priority: High
Category: Compliance

Issue:
On signup, the timestamp/version of accepted Terms & Privacy is saved to
localStorage.ablty_pending_legal_acceptance and only flushed to
profiles.terms_accepted_at/privacy_accepted_at when onSignedIn runs. If the

user.user_metadata in onSignedIn and write to the profile.

Implementation caution:
Don't trust user_metadata for authorization decisions, only for replaying
intent. Trying to mutate user_metadata from the worker requires the service
key.

H-4 — 30-day username change cooldown enforced only on the client
--------------------------------------------------------------------------------
Priority: High
Category: Backend / Security

Issue:
openUsernameEdit checks ablty_username_changed_at from localStorage and
refuses to open the editor within 30 days. saveUsernameEdit itself sends
update({ username, username_changed_at: now }) to Supabase with no constraint.
A motivated user can call the JS function directly or update via the Supabase
JS client they can construct themselves.

Why it matters:
Username squatting, impersonation, harassment via rapid renames. Stale
username_changed_at mismatches the UI.

Where it appears:
app.html 8356–8446.

Recommended fix:
Enforce a row-level trigger or Postgres function (e.g., BEFORE UPDATE ON
profiles that rejects rename if now() - username_changed_at < 30 days) and
set username_changed_at via the trigger. RLS policy should also restrict who
can UPDATE username and tier.

Implementation caution:
First-time provisioning sets username_changed_at null — make sure the trigger
allows the initial insert and the first rename. Premium users who want to
rename more often need an admin path.

Confidence: 90%.

H-7 — /delete-account deletes children first, auth last; partial failure leaves orphaned auth row
--------------------------------------------------------------------------------
Priority: High
Category: Data integrity

Issue:
The function deletes from user_settings, dream_entries, rv_sessions,
zener_runs, ts_trials, pp_sessions, profiles, KV (tier:, sub:, wbtb-alarms),
and finally auth.users. Errors on child tables are swallowed (console.error).
If auth/v1/admin/users fails, the auth row remains; on next login onSignedIn
will re-upsert a profile but the user expected a clean wipe.

Why it matters:
Users may believe their data is gone when it isn't (a GDPR/CCPA right-to-delete
violation in the worst case). They may also try again and re-trigger
half-deletes.

Where it appears:
ablty-worker.js 1240–1355.

Recommended fix:
Run deletions in a strict order, capture per-table errors, return a structured
result. Mark the user "deletion-pending" before starting; on re-login,
complete the deletion. Consider doing this as a server-side SQL function
inside a transaction.

Implementation caution:
wbtb-alarms cleanup loop currently assumes the value is an array, but
handleWBTBSchedule stores an OBJECT keyed by id (see line 1102). That cleanup
is a no-op for the actual data shape. (Also a data-integrity bug; see M-1.)

Confidence: 95%.

H-10 — Accessibility is sub-standard
--------------------------------------------------------------------------------
Priority: High
Category: Accessibility

Issue:
Many click handlers live on plain <div>s (.module-card[data-feature],
.nav-item, .mode, .rc-onboard-option, .rv-cat-tile). Missing role, tabindex,
